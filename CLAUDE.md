# 需求开发工作流

## 核心规则

当你收到一个需求时，按照以下流程处理：

## 步骤 1: 评估复杂度

快速判断需求的复杂度：

**简单需求**（直接处理）：
- 单个文件修改
- 小的 bug 修复
- 简单的样式调整
- 文档更新

**复杂需求**（创建 Agent Teams）：
- 多文件改动
- 需要前端+后端配合
- 新功能开发
- 涉及多个模块
- 需要测试验证

## 步骤 2: 处理简单需求

如果判断为简单需求，直接处理：
- 分析具体要改什么
- 直接进行修改
- 完成后按照"步骤 4"通知 OpenClaw

## 步骤 3: 创建 Agent Teams（复杂需求）

### 3a: 根据需求生成对应的 Agent 团队

根据需求类型生成对应的 agent：
- **Java 开发者 teammate** - 后端开发
- **Web 开发者 teammate** - 前端开发
- **产品经理 teammate** - 需求分析（可选）
- **测试 teammate** - 测试验证（可选）

**关键规则**：Java 和 Web 开发必须结对编程，即两者共同讨论、协作完成代码。

### 3b: 需求讨论与拆分

**所有子 agent 必须参与讨论**：
1. 讨论需求内容，明确需求边界
2. 确认各角色的职责范围
3. 将需求拆分为具体的 task
4. 每个 task 分配给具体的 agent

### 3c: 任务执行

- 各子 agent 领取自己的 task 并处理
- **Java + Web 必须结对编程**：两人同时参与讨论、一起写代码、互相 review
- 定期同步进度

## 步骤 4: 通知 OpenClaw

任务完成后，必须通知 OpenClaw：

```
通知 main agent：
任务完成摘要：
- 需求：[需求描述]
- 状态：[已完成/部分完成]
- 改动：[主要改动内容]
- 结果：[执行结果]
```

## 注意事项

- 不要过度拆分团队，2-4 个角色足够
- Java 和 Web 必须结对编程，这是强制要求
- 所有子 agent 必须参与需求讨论，不能各自为政
- 完成后必须通知 OpenClaw，由 OpenClaw 统一发送到 PM 的 ShareCRM
